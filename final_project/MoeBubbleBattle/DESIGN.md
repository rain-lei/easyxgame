# 《萌泡大作战》项目设计基线

## 1. 项目定位

《萌泡大作战》是一款使用 C++、Visual Studio 和 EasyX 开发的原创 Q 版单人网格闯关游戏。玩法借鉴水泡爆炸类游戏，但不使用商业游戏的名称、角色、地图、图片或音频素材。游戏画面主要由 EasyX 基本图元绘制，原创角色设定图用于美术与报告参考。

玩家在迷宫中移动并放置水泡。水泡倒计时结束后向上下左右释放水浪，能够破坏木箱、触发其他水泡并淘汰敌人。玩家击败本关全部敌人后进入下一关，完成三关即胜利。

## 2. 必做功能

- 开始菜单、操作说明、游戏界面、暂停、过关、失败和通关界面。
- 三个递进关卡，敌人数量、移动速度和地图障碍逐步增强。
- 玩家四方向移动、网格与障碍碰撞、放置水泡。
- 水泡容量限制、2.5 秒倒计时、十字形水浪和连锁触发。
- 固定墙、可破坏木箱、空地三类地图单元。
- 敌人自动寻路/避险；普通敌人接触水浪后被淘汰，第三关多血量 Boss 需要多次命中。
- 木箱被破坏后有概率掉落强化道具。
- 四类道具：水浪范围、同时放泡数量、移动速度、护盾。
- 分数、生命、当前关卡、剩余敌人和强化状态显示。
- 重新开始、返回菜单和安全退出。

## 3. 操作方式

| 操作 | 按键 |
| --- | --- |
| 移动 | `W/A/S/D` 或方向键 |
| 放置水泡 | `Space` |
| 暂停/继续 | `P` |
| 重新开始 | `R`（失败或通关界面） |
| 返回/退出 | `Esc` |
| 界面按钮 | 鼠标悬停并左键点击 |

## 4. 游戏规则

- 四名角色具有差异化初始生命、移速、水泡容量、爆破范围和护盾，具体数值见 `STORY_APPENDIX.md`。
- 已放置的水泡在玩家离开其格子后成为不可穿越障碍。
- 水泡倒计时 2.5 秒，水浪沿四个方向传播。
- 固定墙完全阻挡水浪；木箱被破坏并终止该方向传播；其他水泡会立即被连锁触发。
- 玩家被水浪击中损失 1 点生命并短暂无敌；护盾可以抵消一次伤害。
- 敌人全部淘汰后进入下一关；玩家生命归零则游戏失败。
- 破坏木箱获得 20 分，淘汰普通敌人获得 300 分，淘汰 Boss 获得 1500 分，通过关卡获得额外奖励。

## 5. 面向对象类设计

| 类 | 职责 |
| --- | --- |
| `MoeBubbleGame` | 初始化窗口、维护游戏循环、切换场景、组织各模块协作 |
| `InputManager` | 保存当前帧和上一帧键盘状态，区分按下与持续按住；游戏类同时处理鼠标消息与点击区域 |
| `GameMap` | 管理地图单元、障碍碰撞、木箱破坏和关卡生成 |
| `GameObject` | 可更新、可绘制对象的抽象基类 |
| `Character` | 玩家和敌人的共同基类，封装位置、速度、碰撞盒与存活状态 |
| `Player` | 处理输入、生命、强化状态和放泡能力 |
| `Enemy` | 敌人抽象基类，声明 AI 决策接口 |
| `PatrolEnemy` | 随机巡逻，在路口改变方向 |
| `HunterEnemy` | 根据玩家位置追踪，同时规避危险区域 |
| `BossEnemy` | 多血量首领，集中配置生命并重写受击、得分、AI 与绘制接口 |
| `WaterBubble` | 倒计时、爆炸范围、连锁触发和所有者容量归还 |
| `WaterWave` | 表示短暂存在的水浪格子并完成伤害判定 |
| `PowerUp` | 封装道具种类、移动/拾取状态和绘制 |
| `ItemIconAtlas` | 加载并透明绘制四格道具图集，在地图与 HUD 间复用 |
| `StarParticle` | 命中、破坏和过关时的粒子效果 |

其中 `GameObject`、`Character` 和 `Enemy` 使用虚函数体现运行时多态；`Player`、`PatrolEnemy`、`HunterEnemy`、`BossEnemy` 通过继承复用角色能力；地图、对象容器和游戏状态由 `MoeBubbleGame` 组合管理。

### 5.1 对象职责与协作

- **封装**：`GameMap` 将地图数组 `tiles_` 设为私有数据，只通过 `tileAt`、`isWalkable`、`destroyCrate` 等公开方法访问；`Player` 将生命、速度、水泡容量、爆破范围和护盾封装为私有状态，只能通过 `takeDamage`、`applyPowerUp`、`onBubblePlaced` 等行为修改。调用者不直接改写对象内部数据，状态约束集中在类内完成。
- **继承**：`GameObject` 是统一的可更新、可绘制对象接口；`Character` 继承它并复用位置、速度、碰撞与移动能力；`Player` 和抽象类 `Enemy` 再继承 `Character`；`PatrolEnemy`、`HunterEnemy`、`BossEnemy` 继承 `Enemy`，只实现各自差异化的决策和表现。
- **多态**：`MoeBubbleGame` 使用 `std::vector<std::unique_ptr<Enemy>>` 保存不同派生敌人，通过 `Enemy*` 统一调用虚函数 `draw`、`chooseDirection`、`takeWaveHit`、`scoreValue`、`health`。第三关 Boss 无需让主循环针对具体类型重复编写更新和碰撞逻辑，运行时会自动调用派生类实现。

### 5.2 STL 数据结构与算法

| STL 组件 | 保存或处理的数据 | 选用原因 |
| --- | --- | --- |
| `std::array<TileType, 195>` | 15×13 固定地图格 | 容量编译期确定、连续存储、无动态分配 |
| `std::array<bool, 256>` | 当前帧、上一帧和短按键盘状态 | 虚拟键码范围固定，可按键码 O(1) 访问 |
| `std::vector<WaterBubble>` 等 | 水泡、水浪、道具、粒子和危险格 | 游戏过程中数量动态变化，便于追加、遍历和批量删除失效对象 |
| `std::vector<std::unique_ptr<Enemy>>` | 巡逻、追踪和 Boss 三类异构敌人 | 结合 RAII 自动管理生命周期，并支持基类指针运行时多态 |
| `std::mt19937` 与标准分布 | 地图木箱、道具掉落、AI 方向和粒子参数 | 随机状态独立封装，固定种子便于复现实验和测试 |
| `<algorithm>` | `find`、`count`、`clamp`、`remove_if`、`min_element` | 用标准算法表达查找、边界约束、清理和 AI 评分，减少手写循环错误 |

对象清理采用擦除-移除模式：先由 `std::remove_if` 把失效对象移动到容器尾部，再调用 `erase` 释放尾部元素。敌人容器中的 `unique_ptr` 会随元素删除自动释放派生对象，避免手工 `delete` 和内存泄漏。

## 6. 状态流程

`主菜单 → 操作说明/开始游戏 → 游戏中 ↔ 暂停 → 关卡完成 → 下一关 → 最终胜利`

游戏中玩家生命归零进入失败界面；失败或胜利界面可以重新开始，也可以返回主菜单。

## 7. 地图与界面

- 窗口尺寸：`960 × 720`。
- 地图：`15 × 13` 个格子，每格 `44 × 44` 像素。
- 左侧为游戏地图，右侧为分数、生命、关卡、敌人数量和操作提示。
- 固定墙、木箱、水泡、水浪、角色和道具采用不同高对比度颜色与 Q 版形状绘制。
- 玩家出生点附近预留空地，防止开局被木箱封死。

## 8. 运行检查

- Visual Studio 的 Debug/Release、Win32/x64 均能编译。
- 不存在绝对路径，不依赖本机图片或音频。
- 玩家不能穿墙或穿过已经离开的水泡格。
- 水浪被墙/木箱正确阻挡，连锁水泡只爆炸一次。
- 强化数值存在上限，连续按键不会无限生成水泡。
- 三关均可正常进入、胜利、失败、暂停和重开。
- 至少整理 12 个测试用例，其中包含地图边界、连续按键、连锁爆炸和生命归零等情况。

## 9. 可继续扩展的方向

- 背景音乐和提示音。
- 更多敌人类型与随机地图种子。
- 排行榜或最高分本地保存。

这些内容在现有对象边界上继续扩展，不需要改变主循环的基本结构。
